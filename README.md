# 자동화 데이터 파이프라인 기반 교내 통합 학습·정보 플랫폼


# 1. 과제 주제 (문제 정의)

**과제명**: 자동화 크롤링 파이프라인과 RDBMS 기반 RAG로 구성된 **교내 통합 학습·정보 플랫폼**

**문제 정의**: 교내 공지사항·장학·비교과·취업 정보가 매일 포털에 쏟아지지만, 학생들은 학업으로 인해 주기적인 포털 접속을 놓쳐 '정보 비대칭'과 '기회 상실'을 겪습니다. 동시에 시험 기간에는 강의 자료를 스스로 정리·문제화하기 위한 **개별 학습 보조 도구의 부재**라는 또 다른 문제도 존재합니다. 두 문제 모두 "**교내 데이터를 신뢰성 있게 수집·구조화하고, 학생 개인의 맥락에 맞춰 재배포한다**"라는 동일한 인프라로 해결할 수 있습니다.

**해결 방안**:

1. 파이썬 웹 크롤러로 교내 포털을 일일 자동 수집 → AI 분류기로 카테고리·추천대상 메타데이터 정제 → **RDBMS에 적재** → 카테고리별 **게시판 웹앱**으로 1차 제공.
2. RDBMS에 축적된 검증된 교내 데이터만을 근거로 답변하는 **할루시네이션 최소화 챗봇** 제공.
3. 사용자가 업로드한 강의 자료(PDF 등)를 LLM으로 분석하여 **예상 문제를 자동 생성**하고, 이를 '과목별 문제은행' 테이블에 아카이빙하는 **AI 학습 보조 기능** 제공.

---

# 2. 주제 선정 배경 (이유 또는 동기)

**1) 포털 미접속으로 인한 혜택 누락 경험** 시험 기간이나 과제가 몰리는 시기에는 1주일 넘게 포털을 확인하지 못하는 경우가 많았고, 뒤늦게 접속해 보면 유익한 프로그램이나 장학금 신청 기한이 지나버린 경험이 반복되었습니다.

**2) 챗봇 단일 구조의 한계 인식 → 데이터-우선 아키텍처로 전환** 초기에는 챗봇 형태로 우선 구현할 계획이었으나, 팀 회의 결과 **확장성과 검증 가능성** 측면에서 한계가 분명했습니다. 챗봇 응답만 제공할 경우 사용자는 답변의 출처를 즉각 확인하기 어렵고, 데이터 자체의 품질 검증도 불가능합니다. 따라서 **"학교 데이터를 크롤링 → RDBMS에 적재 → 게시판으로 노출"** 을 1차 산출물로 분리하고, 챗봇은 이 검증된 RDBMS를 참조하는 2차 레이어로 두기로 방향을 전환했습니다. 이 구조는 향후 다른 서비스(모바일 앱, 알림 봇 등)로 데이터 레이어를 재활용할 수 있다는 장점이 있습니다.

**3) 학기 내내 사용되는 '데일리 + 시험기간' 듀얼 리텐션 서비스** 공지 브리핑은 평소에 매일 사용되고, 문제 자동 생성 기능은 시험 기간에 집중적으로 사용됩니다. 두 기능이 **사용 시점이 다르기 때문에 서로 보완적**이며, 학기 전 구간에서 꾸준한 활성 사용자를 확보할 수 있는 구조입니다.

# 3. 과제 수행 계획

본 프로젝트는 회의에서 합의된 **3단계 개발 우선순위**에 따라 다음과 같이 진행됩니다.

#### Phase 1. 데이터 파이프라인 및 게시판 웹앱 구축 (3~5월) — _최우선_

- **크롤러 개발**: 공주대 포털의 일반공지·학사·장학·취업정보 게시판을 타깃으로 파이썬 크롤러를 구축하고, 매일 자정 스케줄러로 신규 글을 수집합니다.
- **AI 메타데이터 정제 파이프라인**: 수집된 원문에 대해 LLM 프롬프팅으로 **카테고리 분류(장학/취업/학사 등)와 추천대상(학과·학년·관심사)** 을 자동 부여합니다.
- **RDBMS ERD 설계 및 적재**: `notices`, `notice_categories`, `users`, `user_interests` 등 핵심 테이블 ERD를 설계하고 일별 적재·중복 방지 로직을 구현합니다.
- **게시판 UI 구현**: FastAPI(백엔드) + Streamlit(프론트) 조합으로 카테고리별 필터링이 가능한 게시판 화면을 제공합니다. 이 시점에서 **챗봇 없이도 단독으로 기능하는 서비스**가 완성됩니다.

#### Phase 2. RDBMS 기반 RAG 챗봇 고도화 (6~8월)

- **검색·답변 파이프라인**: 사용자의 자연어 질문을 RDBMS 쿼리 또는 임베딩 기반 유사도 검색으로 변환하고, 매칭된 공지의 원문·링크를 근거로 답변을 생성합니다.
- **할루시네이션 방지 설계**: 외부 웹 검색을 차단하고 **반드시 RDBMS 내 데이터만 출처(URL 포함)로 제시**하는 프롬프트 정책을 적용합니다.
- **개인화 필터링**: 사용자 프로필(학과·관심사)을 기반으로 답변 우선순위를 조정합니다.

#### Phase 3. AI 학습 보조 (문제 자동 생성) 게시판 추가 (9~10월)

- **자료 업로드 및 추출**: 사용자가 챗봇 인터페이스에 강의 자료(PDF 등)를 첨부하면 텍스트 추출 후 LLM 프롬프팅으로 핵심 개념을 정리합니다.
- **예상 문제 생성 로직**: 객관식·서술형 문제와 정답·해설을 함께 생성하여 `subject_questions` 테이블에 적재합니다.
- **LangGraph 라우팅 에이전트 도입**: 사용자의 질문 의도를 분석하여 **공지사항 테이블을 조회할지, 문제은행 테이블을 조회할지** 결정하는 라우팅 에이전트를 구현합니다. 이 단계에서 챗봇이 단순 Q&A를 넘어 **멀티 도메인 통합 인터페이스**로 확장됩니다.
- **아카이빙 게시판**: 생성된 문제를 과목별로 모아 다른 사용자도 열람·복습할 수 있는 게시판을 제공합니다.

#### Phase 4. 시스템 안정화 및 최종 점검 (11월)

- 크롤러 장애 시 재시도·알림, 중복 적재 방지, RDBMS 인덱스 튜닝 등 운영 안정성 점검.
- 할루시네이션 케이스 회귀 테스트 및 사용자 피드백 반영.
- 산학협력 프로젝트 최종 시연 및 결과 보고서 작성.

# 4. 기대효과 (활용방안)

**1) 정보 비대칭 해소 및 혜택 수혜율 극대화** 일일 자동 큐레이션을 통해 학생들이 장학·비교과·취업 기회를 놓치지 않도록 돕습니다.

**2) 검증 가능한 RAG 인프라 확보** RDBMS 기반 구조 덕분에 모든 답변의 근거 데이터를 직접 조회·감사할 수 있어, **교내 공식 서비스로 채택될 만한 신뢰도**를 확보합니다.

**3) 학기 전 구간을 커버하는 듀얼 유즈케이스** 평시에는 공지 브리핑·Q&A로, 시험 기간에는 자료 기반 문제 자동 생성으로 사용되어 **연중 지속 가능한 액티브 유저 풀**을 형성합니다.

**4) 확장 가능한 데이터 자산화** 정제·구조화된 교내 데이터 자체가 자산이 되어, 향후 모바일 앱, 카카오톡 알림 봇, 학과 단위 맞춤 서비스 등 **2차 파생 서비스의 기반 인프라**로 재활용할 수 있습니다.

# 5. 데이터 확보 및 활용 계획

#### 1) 교내 퍼블릭 데이터 (크롤링 기반 수집)

- **확보 대상**: 공주대 포털 일반공지, 학사공지, 장학공지, 취업정보 및 부서별 모집 글.
- **확보 방법**: Python 크롤링 스크립트를 통해 일일 단위로 제목·본문·게시일·URL을 수집합니다.
- **활용 계획**: AI 메타데이터 정제 후 **RDBMS의 `notices` 관련 테이블에 적재**하여 게시판과 챗봇이 공통으로 참조합니다. 모든 챗봇 응답은 원문 URL을 출처로 함께 반환하여 신뢰도를 보장합니다.

#### 2) 사용자 업로드 학습 자료 (문제 생성용)

- **확보 대상**: 사용자가 직접 업로드한 강의 자료(PDF, 강의 노트 등).
- **활용 계획**: 텍스트 추출·LLM 분석 후 생성된 예상 문제와 정답을 `subject_questions` 테이블에 적재합니다. 원본 자료는 사용자 본인 외에는 접근할 수 없도록 권한을 분리하고, 생성된 문제만 선택적으로 공개 게시판에 공유할 수 있게 합니다.

#### 3) 사용자 프로필 데이터 (개인화 연산용)

- **확보 대상**: 소속 학과, 학년, 관심 키워드(예: 알고리즘, 단기근로, 자격증).
- **활용 계획**: Streamlit UI 최초 진입 시 입력받아 `users`·`user_interests` 테이블에 저장하며, 일일 브리핑 우선순위 산정과 챗봇 답변 개인화의 매개변수로 활용합니다.
